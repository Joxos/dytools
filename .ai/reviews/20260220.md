## 1. Architecture Overview

Affairon is an affair-driven (event-driven) framework with a clean layered architecture:

```
affairs.py          → Domain models (Pydantic)
registry.py         → Dependency graph (NetworkX)
base_dispatcher.py  → Registration API (on/on_method/register/unregister)
dispatcher.py       → Sync emit
async_dispatcher.py → Async emit (TaskGroup)
composer.py         → Plugin loading (entry points + local imports)
fairun/cli.py       → CLI entry point
```

The separation of concerns is sound. Each module has a single responsibility and the dependency flow is strictly top-down.

## 2. Strengths

### 2.1 Type System

- PEP 695 `type` statements for `StandardResult`, `SyncCallback`, `AsyncCallback` — modern and clean.
- Generic `BaseDispatcher[CB]` and `BaseRegistry[CB]` allow sync/async to share logic without `Any` leaks.
- `on()` uses properly bounded TypeVars: `[A: MutableAffair, R: (dict[str, Any] | None)]`.
- `on_method()` correctly uses `F: Callable[..., Any]` — unavoidable since bound/unbound method signatures differ.

### 2.2 API Design

- `on()` vs `on_method()` split is the right call. After exploring 5+ alternatives (inspect, `__qualname__`, `__set_name__`, pending/claim), the explicit API split is the simplest and most reliable.
- `AffairAwareMeta.__call__` running `_bind_affair_methods()` after `__init__` is elegant — no `super().__init__()` footgun.
- `after` parameter for dependency ordering is intuitive and maps well to NetworkX's DiGraph.

### 2.3 Code Quality

- Consistent Google-style docstrings with `Args`, `Returns`, `Raises`, `Post` sections.
- Minimal dependencies, all justified: Pydantic (validation), NetworkX (graph), loguru (logging), packaging (version parsing).
- No `as any` / `@ts-ignore` equivalents — `type: ignore[attr-defined]` only where Python's type system genuinely can't express dynamic attribute stamping.

### 2.4 Testing

- Tests are well-organized by concern: affairs, registry, dispatcher, async, unregister, decorator, affair_aware.
- `conftest.py` provides reusable fixtures (`Ping`, `Pong`, `MutablePing`).
- AffairAware tests cover lifecycle, after-deps, inheritance, staticmethod/classmethod, and mixed scenarios.

## 3. Issues & Concerns

### 3.1 High Severity

**`AffairdError` typo**: The base exception class is named `AffairdError` (extra 'd'). This is part of the public API (`__init__.py` exports it, `__all__` includes it). Renaming later would be a breaking change.

**`emit()` docstring claims `affair_id` and `timestamp` are set, but they aren't**: Both `Dispatcher.emit()` and `AsyncDispatcher.emit()` docstrings state "affair.affair_id and affair.timestamp set" in the `Post` section, but neither implementation sets these fields. The `MutableAffair` model doesn't even define `affair_id` or `timestamp` fields. This is dead documentation from an earlier design iteration.

**`except* Exception` in `AsyncDispatcher` is misleading**: The `except*` clause catches and immediately re-raises. It provides no value — `TaskGroup` already propagates `ExceptionGroup` natively. The `except*` / `raise` is a no-op that adds visual noise.

### 3.2 Medium Severity

**No MRO-aware affair type resolution in `exec_order()`**: When emitting `ChildAffair(ParentAffair)`, only callbacks registered on `ChildAffair` fire — not those on `ParentAffair`. The docstring for `exec_order()` mentions "MRO expansion" but the implementation does a simple dict lookup: `if affair_type not in self._graphs: return []`. This may be intentional (explicit registration only) but contradicts the documentation.

**`_bind_affair_methods` registers on every instantiation without idempotency**: Creating `Handler("a")` then `Handler("b")` registers two bound methods. This is correct by design (documented in tests), but there's no way to unregister a specific instance's methods short of knowing the exact bound method references. If an `AffairAware` instance is short-lived, its callbacks leak in the registry forever.

**`compose_local` swallows exception context poorly**: The method logs the exception via `composer_logger.exception()` then re-raises. But the `except Exception as exc` doesn't chain (`raise ... from exc`), so the loguru traceback and the Python traceback may diverge in chained exception scenarios.

**Plugin deduplication is by name, not identity**: `loaded_plugins` is a `set[str]` of package names. If two different requirement strings resolve to the same package (e.g., `my-plugin>=1.0` and `my_plugin`), PEP 503 normalization may not catch it since `Requirement.name` preserves the original casing/hyphens.

### 3.3 Low Severity

**`_types.py` exports are not in `__init__.py`**: `StandardResult`, `SyncCallback`, `AsyncCallback` are internal types but `StandardResult` was discussed as "要暴露出来" (should be exposed). Currently it is not exported.

**Duplicate dev dependencies**: `pyproject.toml` declares dev deps in both `[project.optional-dependencies] dev` and `[dependency-groups] dev` with slightly different version constraints (`pytest>=8.0` vs `pytest>=9.0.2`). This is confusing.

**`isort` listed in HOW_TO.md but not in dev dependencies**: The project rules specify `isort` for import sorting, but `ruff` handles this via `I` rule. `isort` itself is not installed. This is fine in practice (ruff supersedes isort) but the documentation is outdated.

**No `__all__` in submodules**: Only `__init__.py` has `__all__`. Other modules like `affairs.py`, `exceptions.py` don't declare it. Not blocking but reduces clarity for tooling.

**Guardian callback is a staticmethod on the concrete dispatcher**: `_sample_guardian` is defined on `Dispatcher`/`AsyncDispatcher` rather than being a module-level function or abstract factory. This works but couples the guardian lifecycle to class definition.

## 4. Test Coverage Gaps

### Currently tested (28 tests):
- Affair: custom fields, validation wrapping, frozen immutability
- Registry: after ordering, unregistered after raises, cycle detection, remove
- Dispatcher: single listener, key conflict, on() end-to-end
- AsyncDispatcher: merge results, ExceptionGroup propagation
- Unregister: 3 modes + no-args error
- AffairAware: lifecycle, edge cases, after deps, mixed plain+method, inheritance, override, multi-instance, staticmethod, classmethod, mixed
- Decorator: on() immediate, on_method() deferred, metadata stamping

### Not tested:
- **MutableAffair**: mutable field assignment, extra field rejection (`extra="forbid"`)
- **Dispatcher edge cases**: emit with no listeners (`{}`), listener returns `None` (ignored), non-dict return (should `TypeError`?), multi-layer ordering
- **AsyncDispatcher**: same-layer parallelism verification, single callback raise (not `ExceptionGroup`), key conflict in concurrent results
- **Registry**: `exec_order()` for unregistered type (`[]`), `remove(None, None)` (`ValueError`), cycle rollback preserves prior state
- **merge_dict**: empty dicts, no conflict, single conflict (already tested indirectly via dispatcher)
- **PluginComposer**: all 3 error paths (PluginNotFoundError, PluginVersionError, PluginEntryPointError)
- **fairun CLI**: non-existent dir, missing pyproject, happy path
- **Affair inheritance**: `ChildAffair(Affair)` field inheritance, MRO behavior with dispatchers

## 5. Suggestions

### Short-term (before MVP)
1. Fix `AffairdError` → `AffairError` typo (breaking change now is cheap, later is expensive).
2. Remove stale `affair_id` / `timestamp` references from `emit()` docstrings.
3. Remove no-op `except* Exception: raise` in `AsyncDispatcher.emit()`.
4. Deduplicate dev dependencies in `pyproject.toml`.

### Medium-term
5. Add instance callback cleanup: `AffairAware.__del__` or explicit `unbind()` method to prevent callback leaks.
6. Decide on MRO-aware emit: should `ChildAffair` trigger `ParentAffair` listeners? Document the decision clearly either way.
7. Add `StandardResult` to public exports if it's meant to be user-facing.
8. Fill the medium-priority test gaps listed above.

### Long-term
9. Affair tracing: `affair_id` + `timestamp` metadata injection (the docstrings promise it — either implement or remove).
10. Plugin name normalization per PEP 503.
11. Consider `WeakRef` for `_affair_dispatcher` on stamped methods to avoid preventing dispatcher GC.

---

## 6. Overall Assessment

The project is in solid pre-MVP shape. The core architecture (affair → registry → dispatcher → emit) is clean, well-typed, and tested at the critical paths. The `on()` / `on_method()` API split and `AffairAwareMeta` metaclass are well-designed solutions that emerged from thorough exploration of alternatives.

Main risks are: stale documentation that promises unimplemented features (`affair_id`, `timestamp`, MRO expansion), the `AffairdError` typo baking into the public API, and callback lifecycle management for `AffairAware` instances.

The codebase is disciplined — consistent style, proper docstrings, clean git history. Ready for the next phase of test hardening and edge case resolution.
