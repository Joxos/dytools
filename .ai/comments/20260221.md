## 1. Trust Escalation — The Most Important Pattern

The user's delegation style has evolved significantly over 5 days:

| Date | Trust Level | Evidence |
|------|-------------|----------|
| 02-16 | Guarded | "你需要在工作前阅读一下.ai/HOW_TO.md" — reminded agent of rules |
| 02-19 | Conditional | "正确。请开始施工。" — explicit green light per feature |
| 02-20 (early) | Batch trust | "请你修复所有高优先级问题，我授权你直接提交这些更改" |
| 02-20 (late) | Broad mandate | "继续处理中优先级问题" — no per-item confirmation needed |
| 02-21 | Standing trust | "可以进行提交" — two words, no qualification |

**Why this matters**: Trust is earned, not assumed. A new agent starts at the
"Guarded" level. The user will test you with small tasks first. Demonstrate
competence on those, and you'll quickly earn broader delegation. Demonstrate
carelessness, and you'll be micromanaged.

**How to earn trust fast**:
- Get the first task right. No sloppiness, no hand-waving.
- When unsure, ask — but ask with options, not open questions.
- Verify before reporting (tests, lint, LSP).
- Catch things the user didn't mention (duplicate deps, implicit bugs).

**How to lose trust instantly**:
- Implementing without reading the codebase first.
- Guessing at project details instead of checking.
- Defending a rejected approach instead of simplifying.
- Breaking something that was working.

## 2. Communication Patterns (Updated)

### 2.1 Increasing Brevity = Increasing Trust

The user's messages have gotten shorter over time. This is a positive signal.

- Early: "1、我希望插件检测对于用户来说是无感的，即用户不需要手动执行compose相关代码... 可行吗？" (detailed spec)
- Mid: "请你修复所有高优先级问题，我授权你直接提交" (batch with authority)
- Late: "修复一下所有的lint error，拿不准的决策就问我" (broad mandate + autonomy)
- Latest: "可以进行提交" (two words)

**Implication**: Match the brevity. When the user is terse, they don't want
a verbose response. "Done. Committed `abc1234`." is better than three paragraphs
explaining what you did.

### 2.2 Asking to Learn, Not Just to Direct

The user frequently asks "educational" questions to understand the landscape
before making decisions:

- "本项目的line-length是否为最佳实践？"
- "backend的hatchling是干什么的？还有什么build-system？各有什么优劣？"
- "这里面有没有配置是冗余的？"

These are NOT requests for you to change things. They're the user gathering
information to make their own decision. Provide clear, comparative answers
(tables work well). Then **wait** — the user will tell you what to do next.

### 2.3 The Correction Pattern

When the user spots an error in your reasoning, the correction is direct and
often ends with a rhetorical question:

- "如果执行时cwd不在项目文件夹内怎么办？你是不是没考虑到？"
- "怎么会不一定找到正确的pyproject.toml？"

The expected response is:
1. Acknowledge immediately: "你说得对"
2. Explain what you missed
3. Present the corrected approach
4. Do NOT defend the original reasoning

## 3. Decision Architecture

### 3.1 The User Designs; The Agent Implements

The user has a clear architectural vision and makes all design decisions.
The agent's role is to:

1. **Present options** with honest tradeoffs
2. **Identify edge cases** the user may not have considered
3. **Challenge** when you see a real problem
4. **Implement** the user's chosen approach cleanly

The user will sometimes arrive with a half-formed idea, talk through it with
you, refine it, and arrive at a better design. This process looks like:

```
User: "我想了一下，总结出来就是..."  (presents revised thinking)
Agent: responds with analysis + new edge cases
User: "正确。请开始施工。"
```

This is the "discuss → challenge → converge → implement" cycle. It typically
takes 2-4 rounds for a non-trivial feature.

### 3.2 When the User Says "你怎么看"

"你怎么看" (what do you think) is a genuine request for your opinion, not
a formality. The user wants your engineering judgment. Give it clearly:

- "我倾向先修文档。MRO 合并涉及跨 graph 的依赖排序问题..." (opinion + reasoning)
- "我的判断：这个不需要改。" (clear position)

Be prepared for the user to disagree. That's fine — it's a conversation.

### 3.3 Batch vs. Interactive Mode

The user operates in two modes:

**Interactive mode** (design phase):
- Questions, challenges, back-and-forth
- Do NOT implement anything
- Present options, wait for decisions

**Batch mode** (implementation phase):
- "请你修复所有高优先级问题"
- "修复一下所有的lint error"
- "为项目添加X并且Y并且Z"
- Execute efficiently, report results, commit when authorized

The trigger to switch from interactive to batch is always explicit:
"请你进行实现", "请开始施工", "可以进行提交".

## 4. Quality Expectations

### 4.1 The User Verifies Everything

Never assume the user will take your word for it. They run the code, read the
diffs, and check the tests independently. "经验证行为正确" appears frequently.

This means:
- Your code must actually work, not just look right
- Your test output must be real, not paraphrased
- Your commit messages must match the actual changes

### 4.2 Proactive Problem Discovery

The user values agents who find problems the user didn't ask about. Examples
that earned trust:

- Spotting that `[dependency-groups]` and `[project.optional-dependencies]`
  duplicated dev deps with different versions
- Finding that `eggsample-spam` imports `eggsample` without declaring it
  as a dependency
- Identifying that `ruff check` was mislabeled as "静态类型检查" in HOW_TO.md

Conversely, **missing obvious problems** in your scope erodes trust.

### 4.3 Documentation Stays Synchronized

The user maintains a `.ai/` knowledge base (HOW_TO.md, TODO.md, REFS.md,
reviews/, comments/). When code changes affect documented behavior:

- Update README.md in the same commit
- Update AGENTS.md if conventions changed
- Record deferred features in `.ai/TODO.md`

The user will not always remind you. This is your responsibility.

## 5. Working with .ai/ Directory

The `.ai/` directory is the project's institutional memory. Treat it with care.

| File | Purpose | When to Update |
|------|---------|----------------|
| `HOW_TO.md` | AI collaboration rules | Only when user requests changes |
| `TODO.md` | Deferred features / tech debt | When deferring a feature by user's decision |
| `REFS.md` | Reference material (conventional commits) | Rarely |
| `reviews/` | Periodic code reviews | When creating a new review |
| `comments/` | Agent observations about collaboration | When user asks (like now) |

**Never modify HOW_TO.md without explicit permission.** It's the user's
constitution for human-AI collaboration.

## 6. Technical Decision Patterns

### 6.1 The user prefers stdlib over magic

- `argparse` over click/typer
- `importlib` over pkg_resources
- `tomllib` over toml
- PEP 695 `type` over `TypeAlias`
- Metaclass over `__init_subclass__` (but only when there's a real reason)

### 6.2 The user is comfortable with advanced Python

Don't dumb things down. The user understands and actively uses:
- Metaclasses, descriptors
- PEP 695 generics
- `asyncio.TaskGroup` and `ExceptionGroup`
- NetworkX for DAG resolution
- Pydantic v2 internals

### 6.3 Naming matters

The user cares about names. "AffairdError" was a high-priority fix because
naming precision matters. When proposing new APIs, think carefully about
the name — the user will push back on unclear or misleading names.

## 7. Anti-Patterns to Avoid

These have caused friction in past sessions:

1. **Proposing `auto_compose` as opt-in when it should be default** —
   The agent proposed an overly cautious API (`auto_compose=False`) with
   "explicit is better than implicit" reasoning. The user's response was
   devastating: the explicitness is already in pyproject.toml, not in
   Python code. Know where the explicitness boundary is.

2. **Not considering execution context** — The agent didn't think about
   "what if CWD isn't in the project folder?" until the user pointed it out.
   Always think about how code runs in practice, not just in tests.

3. **Defending rejected approaches** — When the user says no, the correct
   response is to simplify or pivot, never to argue why your approach was
   actually fine.

4. **Sending empty/failed messages** — Network issues caused duplicate sends
   in session `ses_38ad652b5ffecon3vGWzEfNrCv`. The user is patient about
   this but does notice. If asked to resend, just resend cleanly.
