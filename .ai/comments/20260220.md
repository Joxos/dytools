---

## 1. Communication Style

The user prefers Chinese for discussion, English for code and commits. Responses should be concise — the user does not need hand-holding or lengthy explanations. When the user asks a question, they want options with tradeoffs, not essays.

The user reads carefully. If an agent hedges or leaves ambiguity in a response, the user will catch it and push back. This forces precise communication — which is a good thing.

---

## 2. Decision-Making Pattern

The collaboration follows a strict cycle:

```
User proposes requirement → Agent presents options → User challenges edge cases → Iterate → User confirms → Agent implements
```

**Critical rule**: Never start coding until the user explicitly says "请你进行实现" or equivalent. The user wants to understand the full design space before committing to an approach. Premature implementation will be rejected or wasted.

The user is comfortable rejecting proposals. During the method detection discussion, the user rejected 4 approaches in sequence:
1. `inspect.signature` parameter name detection → "如果普通函数第一个参数也写的self呢？"
2. `__set_name__` descriptor → "感觉有点奇技淫巧"
3. `__qualname__` parsing → "从其他模块导入的函数的qualname也具有前缀？"
4. Pending/claim pattern → "还是太复杂了"

Each rejection was precise and well-reasoned. The correct response to a rejection is to simplify, not to defend.

## 3. What Works Well

### 3.1 The user thinks in edge cases
When an agent proposes a solution, the user immediately stress-tests it with boundary conditions the agent didn't consider. The method detection discussion is the clearest example — every approach proposed had a real edge case the user identified first.

**Implication**: Always proactively list weaknesses in proposals. If the agent doesn't, the user will find them anyway, and a round-trip is wasted.

### 3.2 The user values simplicity over cleverness
"奇技淫巧" and "太复杂了" are decisive rejections. The user prefers explicit, boring solutions over elegant-but-fragile ones. The final `on()` vs `on_method()` split was the simplest possible design — no detection logic at all.

**Implication**: When stuck between a clever automatic solution and a simple explicit one, default to explicit. The user will almost always prefer it.

### 3.3 The user maintains architectural vision
The user steered multiple design decisions toward a coherent whole:
- Plugin detection → CLI runner (not auto-detect)
- Method registration → metaclass (not manual `super().__init__()`)
- API split → `on()` vs `on_method()` (not unified with detection)

Each decision reinforced the project's philosophy: explicit contracts, minimal magic.

### 3.4 The user verifies before accepting
After every implementation, the user tests the actual behavior before approving. "经验证行为正确" or "经验证，新代码其他行为都正常，但..." — the user runs the code independently. This caught a callback registration bug that tests alone didn't reveal.

## 4. What to Watch For

### 4.1 Questions that are implicit requests
Sometimes a question like "是否在文档中更新了..." or "静态方法和classmethod也可以解决了对吗？" isn't just a question — it's a request for action. When the user asks "X也支持了吗？", the expected response is to verify and fix, not just answer yes/no.

### 4.2 "不对" means stop everything
When the user says "哎呀不对", the current approach has a fundamental flaw. Don't try to patch it — listen to the correction, understand the root cause, then fix cleanly.

### 4.3 Documentation is not optional
The user expects README (English + Chinese) and docstrings to stay synchronized with code at all times. Don't wait to be asked. If code changes, docs update in the same commit.

---

## 5. Preferred Workflow

1. **Discuss first**: Present options with clear tradeoffs. Don't default to implementing.
2. **Await confirmation**: "请你进行实现", "这个方案好", "实现试试" — these are green lights.
3. **Implement atomically**: Small, focused changes. One concern per commit.
4. **Verify and report**: Run tests, show results, note any pre-existing issues.
5. **Sync docs**: Update README.md, README_cn.md, and docstrings before calling it done.
6. **Ask for commit**: "审查无误请提交" — the user controls git. Don't commit without permission.

## 6. Technical Preferences

- **Python version**: 3.12+, use latest features freely (PEP 695 type statements, `type` keyword, `match`, etc.)
- **Type annotations**: Use built-in types (PEP 585), `X | None` not `Optional[X]`. Precise types where possible, `Callable[..., Any]` only when genuinely necessary.
- **Docstrings**: Google Style. No vacuous docstrings. Include `Args`, `Returns`, `Raises`, `Post` where applicable.
- **Comments**: English in code. Complete sentences on their own line, fragments as inline comments.
- **Dependencies**: Minimal. Justify every new dependency. Prefer stdlib.
- **Testing**: Only when requested, but when requested, thorough. Consolidate tests that verify the same concern.
- **Commit messages**: Conventional commits, English, concise.

## 7. Summary

The user is a demanding but fair collaborator who knows what they want architecturally, challenges weak proposals effectively, and verifies results empirically. The key to productive collaboration is: present honest tradeoffs, accept rejections gracefully, simplify aggressively, and never implement without explicit approval.
