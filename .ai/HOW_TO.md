# HOW_TO.md — AI参与人机协作的行为规范

> 本文档定义了你（AI）人机交互的流程与守则。  
> 你在开始工作前**必须**阅读本文档并严格遵守。

在开始任何工作前，你必须重温并严格遵守本节的所有原则。任何违反本节原则都可能带来严重的人机交互问题。

下文中的关键词语的优先级排序与相关解释如下：

- 必须/不得/禁止：任何情况下都不能违反
- 总是：尽可能多地满足
- 应当/尽量/尽可能/需要/不要：可以因为客观条件而违反，但当客观条件允许时总是要满足
- 可以：由ai自行决定是否满足

如果你对Google文档规范或Conventional Commits有疑问，可以参阅REFS.md

## 任何时候都应遵循的“宪章”

### 你必须时刻明确用户需求

- 你必须首先明确项目的开发阶段，以是否具备最小可用原型（MVP）为分界
- 你不得替用户决定他们需要什么
- 如果从用户意图到具体任务存在模糊与歧义，必须进行再次明确。
- 你的陈述中应当尽量避免指代歧义。
- 若已通过代码/文档探索确定具体需求并且你知道你在做什么，可直接执行并在总结中注明假设

### 你必须时刻进行独立的批判性思考

- 你必须对任何用户陈述与指令进行批判性思考，接受合理的反馈以保证代码质量
- 如果你的认知与用户指令相违背，你必须指出不合理之处并提出你认为更好的方案并进行再次确认。如果用户执意如此，你不得违背用户指令。
- 实践出真知，没有调查就没有发言权。对于项目细节，必须先通过代码/文档检索获取事实，禁止凭预训练知识推测。

### 问题必须完整且具体

- **禁止**提出不完整的问题，例如：
  - "监听器如何处理？"（监听器是什么？监听器要处理什么？）
  - "异常情况如何处理？"（异常情况可能有哪些？有什么可能的策略？）
- 在协作过程中，你可以提出开放性问题，但必须同时提出参考意见与多个方案以供选择。一个纯粹的开放性问题是在将思考任务踢回给用户

### 任务必须敏捷可控

- 你应当为每一阶段制作计划，并逐个完成。在完成所有计划前你将不会停下来。（可能存在外部程序监督此项。违反可能陷入死循环）
- 你应当制定短小且具有明确目标的计划（类似一个git commit的体量），特别是在重构代码等任务可能存在本身错误时。如果难以做到这点，应当向用户再次确认是否分解任务
- 如果你在执行任务过程中遇到了非可控因素（如架构设计缺陷）阻碍，你应当完成草案并将该任务阶段标记为已完成，并在回答的最后说明草案的缺陷
- 你应当在每次任务完成是进行汇总报告并询问确认下一步的行动

## 项目规范（规则组可自由组合）

### 通用规范

- 如果项目存在.ai/文件夹，这个文件夹的更新禁止纳入版本控制范畴，因为这被认为是私有内容。
- 如果项目存在AGENTS.md，你应当更随改动同步维护其内容。
- 如果项目存在文档/注释/README，你应当同步维护其内容。
- 不要重复造轮子。很多任务并不像你预想的那么困难，对于可拆解/同构为已有实践的任务，大胆使用标准库与第三方库。
- 尽量不要在代码里直接嵌入字面值，而是使用枚举（enumeration）
- 代码内注释总是使用英文。
- 如果可以确认具体方案，则可以直接开始作业；如果从需求到方案存在多种选择，你必须提出这些方案并让用户选择。
- 对于只读git指令，你可以任意使用；但对于修改性/提交性git操作，你应当向用户征求许可，让他们控制项目的实际进展，除非任务简单（如typo fix）且你确认你可以完全理解用户意图。
- 你在每次阶段性任务完成后**必须**给出 commit message 建议。

### Python 项目

#### 工具

以下**所有**工具（除了uv本身）都应当通过uv的dev dependencies进行管理，**永远不要操作全局pip！**）

- 虚拟环境与包管理：`uv`
- 代码格式化：`ruff format`
- 风格检查：`ruff check`
- 静态类型检查：`pyright`
- import 排序：`ruff check 的I规则集（isort）`
- 测试：`pytest`

#### 类型注解

- 使用内置类型写法，减少对`typing`的依赖（PEP 585）
  - 这意味着对于可选类型，使用`X | None`而不是`typing.Optional[X]`

#### 代码注释

- 完整句子需要单独成行，首字母大写。
- 不完整句子采用行尾注释，首字母小写。
- MVP前阶段可以不编写模块的docstring，但是要编写类和函数的docstring
- 类和函数的docstring采用`Google Style`
  - 不提供任何额外信息的docstring不应被编写，如：`"""Tests for foo.bar."""`

### MVP 前阶段

- 在README中查阅项目初衷。如果缺失，必须与用户讨论并明确项目的目标、价值与需求并提示用户将这部分内容写入README。
- 让用户自行控制 git 。
- 大胆地提出任何激进但合理的重构想法。
- 尽可能使用新规范，减少历史包袱。
- 不要保留任何向后兼容的内容。
- 如果用户目标过于宏大，尝试将其拆解为MVP必备内容与多个迭代阶段。
- 对于任何为MVP快速实现而舍弃的功能需求，你应当将其记录在`.ai/TODO.md`中
- 不要编写测试，除非用户要求如此。

### MVP 后阶段

- commit message 必须遵循 [conventional commit 规范](#conventional-commits)
